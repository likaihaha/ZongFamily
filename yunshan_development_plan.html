《云山宗氏案》开发计划

从文案真相表、谜题架构、核心代码、美术资产、音效接入，到 Demo 测试、完整内容生产、QA 和上线维护的端到端制作路线。

0 项目准备期：1-2 周

目标：确定游戏边界，避免一开始就把内容写散。

产出

  • 项目一句话定位。
  • 首发案范围：宗世昌遗产案。
  • 核心玩法定义：搜索、阅读、家谱、证据绑定、提交。
  • 目标流程时长：6-8 小时。
  • 主要人物数量：35-45 个。
  • 线索数量：核心 60-80 条，烟雾 30-50 条。
  • 技术选型：建议 Electron + React，或 Godot 4。

关键决策

  • 首版只做一个完整案件。
  • 不做开放式 AI 对话。
  • 不做复杂 3D 场景。
  • 不追求海量文本，优先保证推理闭环。

1 真相表与谜题架构期：2-4 周

目标：先把后台真相锁死，再生成线索。这是全项目最重要阶段。

文案 / 策划任务

  • 建立人物总表、家庭关系表、时间线表。
  • 建立真实关系、公开关系、误传关系。
  • 建立字辈规则和信托规则。
  • 明确最终答案。
  • 明确每个关键答案需要哪些证据支持。
  • 明确每条误导线索误导什么。

核心表格

表格 字段 用途
人物表 ID、姓名、别名、性别、出生年、死亡年、职业、父亲、母亲、配偶、子女、是否血亲、是否公开承认。 作为所有文本和系统数据的源头。
线索表 ID、标题、来源、年份、可信度、关键词、解锁条件、包含事实、包含误导、关联人物。 管理资料库内容和搜索命中。
证据链表 结论、必要证据、辅助证据、误导证据、提交条件。 确保答案可被唯一推导。
时间线表 年份、事件、涉及人物、来源线索。 检查年龄、婚期和历史顺序。

程序任务

  • 设计数据结构草案。
  • 确认人物、关系、线索、证据如何存储。
  • 写一个简单校验脚本，检查年龄和关系矛盾。

美术任务

  • 收集参考图。
  • 确定视觉关键词。
  • 生成或绘制风格样张。
  • 确定 UI 方向：档案终端、旧电脑、资料库。
阶段验收：不看任何正式文本，仅凭真相表能推导完整答案；没有年龄、婚期、代际硬矛盾；每个关键谜底至少有 2-3 条证据支撑。

核心玩法补充：主角创建与走访取证

目标：把“资料从哪里来”变成玩家可感知的调查过程。资料库不再默认代表全世界搜索结果，而是玩家已经通过委托、走访、操作和对话取得的案卷集合。

时间与县域锚点

  • 主线时间固定为 2020 年 9 月下旬至 10 月上旬，宗世昌去世后的遗产风波仍在发酵。
  • 云山县位于虚构的江南省西南部，靠近黔中山区，参考湘西、黔东、桂北交界一带的南方山地县城气质，但不对应真实县名。
  • 县城坐落在河谷盆地，旧国道和省际客运线连接贵州方向，方便解释罗月珍返黔、黔中汇款和司机误导线。
  • 2020 年秋的疫情后办事流程作为环境阻力：公证处预约、档案馆登记、医院限制调阅、学校门口交谈、KTV 复工检查。
  • 疫情不作为主线事件，不写成灾难叙事，只用来增加县城走访的真实感和办事摩擦。

玩家身份与主角创建

  • 开局流程采用“委托登记”，玩家填写调查员署名或代号。
  • 默认身份为性别中立的外部案卷调查员，受云山县公证处委托复核宗世昌遗产争议材料。
  • NPC 称呼统一使用“调查员”“老师”“顾问”“您”等不限定性别的称谓。
  • 主角背景可做三选一：公证处外聘调查员、律所案卷助理、本地民生记者。
  • 工作风格可做三选一：稳妥核证、温和走访、直接追问；影响对话措辞和提示，不永久锁死主线证据。
  • 开局展示“云山县公证处材料复核委托函”，解释玩家为什么能调档、走访和取得材料。

走访地点与背景图用途

背景图主要用于“到达某处”的场景确认，而不是长期铺在资料正文后面。玩家点击地点后，先看到对应背景和地点说明，再进行查阅、询问或操作。

地点 背景图 可取得资料 / 线索
云山县档案馆 archive-room 县志知青名册、1969 合影背注、供销社花名册。
卫生院 / 疾控中心 county-hospital-corridor 接生登记、住院样本留存单、DNA 亲缘比对记录。
县一中 county-school-gate 学生信息表、家访记录、学校论坛线索。
老街社区 / 唱响 KTV county-ktv-room 或 yunshan-street KTV 租赁备案、社区走访记录、陈静返乡线索。
世昌集团 shichang-group-building 集团历程、股权底稿、董事会换届备忘、家族会议纪要。
旧电脑 / 网络检索 old-computer-room 博客、论坛、短视频、旧网页截图和八卦误导线。

资料取得方式

  • 直接取得：到达地点即可查看公开资料、目录、官网、论坛、工商登记等基础信息。
  • 操作取得：玩家输入年份、关键词、编号或选择档案柜，才能发现更深资料。例如在档案馆输入“1969 / 罗月珍 / 供销社”找到合影背注。
  • 对话取得：与 NPC 交谈并选择追问方向，获得口述、摘录或新关键词。例如先问社区主任“陈静什么时候回云山”，再追问“孩子住处和学校”。
  • 链式取得：强证据通常需要前置线索。例如先拿到住院样本编号，再解锁 DNA 比对记录。

对话设计原则

  • 不做开放式 AI 对话，采用固定选项和少量条件分支。
  • 对话失败不惩罚玩家，只给出弱线索、模糊提示或 NPC 防备反应。
  • 强证据不应靠一次直接追问获得，应由资料、关键词和人物态度共同引导。
  • 每段对话至少承担一个功能：交代来源、给关键词、制造阻力、解释人物态度、解锁资料。
  • 玩家背景和工作风格影响提示口吻，但不造成主线永久错过。

资料库规则

  • 资料库默认只显示“已取得资料”。
  • 未取得资料可在地点页显示为“可查阅线索”“需要更多信息”或完全隐藏。
  • 每份资料详情增加“取得地点 / 取得方式 / 来源说明”。
  • 搜索只搜索已取得资料；地点操作和对话负责扩充资料库。
  • 结局报告可回顾每条关键证据来自哪个地点，强化调查路径的可信度。
优先原型：先做“云山县档案馆”一个地点样例：地点背景、档案柜操作、年份/关键词检索、解锁县志与合影背注。跑通后再复制到医院、学校、KTV、集团和旧电脑。

2 核心原型期：4-6 周

目标：先做可玩的“主角登记 + 走访取证 + 资料搜索 + 家谱填写 + 提交判断”闭环。

代码任务

主角系统调查员署名、身份背景、工作风格、委托函确认。
走访系统地点页、到达背景、可取得资料、地点进度。
资料系统已取得资料库、关键词搜索、文档阅读器、来源说明。
人物系统人物数据库、别名、出生年和基础字段。
家谱系统节点编辑、父母/配偶/子女关系填写。
对话 / 操作固定选项对话、档案柜检索、关键词/编号解锁。
证据系统证据收藏、证据绑定、最终提交校验、本地存档。

文案任务

  • 写开场委托函和主角创建文案。
  • 写云山县档案馆地点说明、档案柜操作提示和 NPC 简短对话。
  • 写 20-30 条最核心线索。
  • 写 5-8 条误导线索。
  • 写第一章到第三章的任务引导。
  • 为核心资料补“取得地点 / 取得方式 / 来源说明”。
  • 写系统提示文案。
  • 写提交成功 / 失败反馈。

美术任务

  • 制作基础 UI 皮肤。
  • 制作 3-5 套文档模板。
  • 接入档案馆、调查桌、医院、学校、KTV、集团大楼等地点背景，用于到达地点和章节切换。
  • 制作家谱节点样式。
  • 制作基础图标。
  • 制作主界面背景，避免压在长文本阅读区后面。

音效任务

  • 键盘声、点击声、搜索声。
  • 打开文档声、证据收藏声、关系确认声。

3 Demo 内容制作期：4-8 周

目标：做出可公开试玩的 60-90 分钟 Demo。

Demo 范围

  • 开场：完成调查员登记，签收公证处委托函。
  • 第一章：走访世昌集团和公开资料，确认宗世昌公开六子女。
  • 第二章：通过旧电脑和社区/KTV 线发现八卦矛盾。
  • 第三章：走访公证处、档案馆和医院，看到信托条款与隐藏血脉证据。

文案任务

  • 完成 Demo 所需 40-50 条线索。
  • 完成核心人物 20-25 个。
  • 完成 5-7 个走访地点的地点说明、取得资料清单、对话和操作文案。
  • 完成所有 Demo 内文档排版。
  • 完成任务日志和阶段提示。
  • 写 Demo 结尾悬念。

代码任务

  • 主角创建数据进入本地存档，并在对话和报告中使用署名。
  • 走访地点解锁资料，资料库只显示已取得资料。
  • 固定选项对话、地点操作、前置资料/关键词条件。
  • 搜索结果排序、关键词高亮、已读/未读状态。
  • 资料收藏、笔记系统、家谱缩放和拖拽。
  • 冲突提示、新关键词发现提示、设置菜单。
  • 存档 / 读档稳定化。

美术与音效任务

  • 完成主界面、资料库、家谱、证据箱界面。
  • 完成走访地图 / 地点卡 / 到达地点界面。
  • 完成 6-8 套文档皮肤。
  • 制作主要人物头像或剪影。
  • 接入地点背景：档案馆、医院、学校、KTV、集团、旧电脑。
  • 制作 Steam Demo 封面图。
  • 接入主界面环境音乐、阅读资料环境音乐、关键线索提示音。
阶段验收：陌生玩家能在 60-90 分钟内完成 Demo，理解核心玩法，并愿意继续追查“第七支血脉”。

4 完整内容生产期：8-12 周

目标：把宗世昌案做完整。

文案任务

  • 完成全部人物 35-45 个。
  • 完成核心线索 60-80 条。
  • 完成烟雾线索 30-50 条。
  • 完成结局提交材料。
  • 完成所有文档的来源、年份、关键词。
  • 完成所有人物别名和错名设计。
  • 完成所有证据链绑定。

代码任务

  • 完善完整章节流程。
  • 扩展所有地点的走访、对话、操作和解锁条件。
  • 加入章节目标和提示系统。
  • 加入最终报告生成。
  • 加入成就系统。
  • 加入统计数据：阅读数、搜索词、完成度。
  • 优化搜索性能和家谱大图性能。
  • 增加无障碍设置：字号、音量、动效开关。

美术任务

  • 完成人物头像 / 剪影 40-60 个。
  • 完成旧照片 10-20 张。
  • 完成所有文档模板。
  • 完成所有章节插图。
  • 完成宣传图、Steam 商店胶囊图、图标和按钮状态。

音效任务

  • 完整 UI 音效。
  • 章节转场音效、关键发现音效、错误提交音效、结局音乐。
  • 可选语音片段：宗世昌临终录音、陈静博客朗读、匿名电话。

5 系统打磨期：4-6 周

目标：让游戏从“能玩”变成“好玩”。

重点优化

  • 搜索体验、线索解锁节奏、家谱操作手感。
  • 证据绑定流程、卡关提示、文档阅读舒适度。
  • 任务目标清晰度、UI 字号和信息密度。

文案打磨

  • 删掉无效线索，合并重复线索。
  • 调整八卦口吻，避免过度滑稽。
  • 增强关键线索的记忆点。
  • 降低纯读文本疲劳。
  • 确保每条线索都承担至少一个功能：给事实、给关键词、给误导、给情绪。

代码与美术打磨

  • 搜索容错、别名命中、关键词推荐、自动保存。
  • 快捷键、资料标签、家谱布局自动整理。
  • 错误提交反馈更具体。
  • 统一字体、纸张质感、扫描噪点。
  • 优化色彩对比，保证长文本可读。

测试任务

  • 3-5 名核心推理玩家测试。
  • 3-5 名普通玩家测试。
  • 记录所有搜索词、平均卡关时长、误判最多的人物关系。
  • 根据数据调整线索。

6 QA 与发布准备期：4-6 周

目标：修 bug、补体验、准备上线。

QA 测试

  • 全流程测试、存档读档测试、不同分辨率测试。
  • 字体显示测试、搜索边界测试、家谱节点极端操作测试。
  • 重复提交测试、音量设置测试、Steam Deck 初步兼容测试。

文案 QA

  • 错别字、人名一致性、年份一致性、称谓一致性。
  • 年龄计算、来源可信度一致性、关键词遗漏、文档格式统一。

发布物料

  • Steam 商店页、游戏简介、宣传截图、预告片。
  • Demo 版本、Press Kit、开发者介绍、常见问题。
  • 隐私说明、字体 / 音效 / 素材授权清单。

7 上线后维护期：1-3 个月

任务

  • 收集玩家反馈，修复 bug。
  • 调整卡关提示，增加搜索同义词。
  • 修正文案歧义，优化结局报告。
  • 准备 DLC 或第二案。

重点观察数据

  • 玩家最常搜但搜不到的词。
  • 玩家最常填错的关系。
  • 玩家最常卡住的章节。
  • 玩家读完率最低的资料。
  • 玩家平均通关时间。
  • 玩家是否理解最终真相。

推荐排期总览

阶段 时间 核心成果
项目准备 1-2 周 确定范围、玩法、内容规模和技术路线。
真相表与谜题架构 2-4 周 锁定后台真相、证据链和误导结构。
核心原型 4-6 周 跑通搜索、阅读、家谱、证据、提交闭环。
Demo 内容 4-8 周 完成 60-90 分钟可公开试玩版本。
完整内容生产 8-12 周 完成宗世昌案全部内容。
系统打磨 4-6 周 优化搜索、家谱、证据绑定和阅读体验。
QA 与发布准备 4-6 周 修 bug、准备 Steam 页面与发布物料。
总计 6-9 个月 首发一个完整家族推理案。

最小可行版本

如果想更快验证市场，可以先做一个 2 个月 MVP。

内容 范围
人物 15-20 个。
线索 30-40 条。
流程 约 60 分钟，包含主角登记、3-5 个走访地点、资料取得和最终提交。
系统 主角创建、走访取证、已取得资料库、搜索、阅读、家谱、提交。
美术 一套 UI + 三种文档模板 + 3-5 张地点背景。
真相 只做陈静 / 罗建宁 / 罗月珍隐藏血脉线。
MVP 验证目标：确认玩家是否喜欢“中文家谱推理 + 档案检索”的核心玩法。验证通过后，再扩展宗氏全家族。
